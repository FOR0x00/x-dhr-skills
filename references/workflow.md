# 流程审批模块

## 当前范围

根据用户输入识别可发起的流程模板，得到流程模板 `ID`，读取对应表单的可填写字段，匹配用户提供的信息，并使用 `UserAsk`
补全匹配失败的必填字段

## 发起审批

### 执行流程

#### 1. 解析流程模板

调用 `get-workflow-template-list` 工具， 获取当前用户可发起的模板。

并按以下优先级匹配：

1. `templateId` 完全一致
2. 模板名称完全一致
3. 用户描述与模板名称语义匹配

只在唯一候选明显成立时自动选择。零个候选时告诉用户没有找到可发起流程；多个候选时使用 `UserAsk` 展示 2 至 5 个最相关候选

确认后保存 `templateId` 后续字段补全期间不要重复查询或更换流程，除非用户修改目标。

#### 2. 获取模板表单定义并建立草稿

调用 `get-workflow-template-form` 工具，并传入已确认的 `templateId`，获取模板对应表单定义。

识别用户输入，匹配表单字段

#### 3. 确认草稿

所有必填字段合法后，展示：

- 流程模板名称。
- 已填写字段的“字段标题：显示值”。
- 仍为空的可选字段数量。

询问用户是否确认发起。不要在确认摘要中展示内部字段 ID；提交工具仍必须使用字段 ID。

#### 4. 发起流程

只有用户确认后才调用语义等价于 `create-workflow-by-template` 的工具

除非用户要求，不要尝试重试调用

### 错误边界

- 工具报权限错误：说明当前账号无权查询或发起，不要绕过权限查找其他接口。
- 流程模板在补全期间失效：丢弃草稿中的目标 ID，重新查询流程列表。
- 表单 schema 发生变化：以最新 schema 重新校验，保留仍可唯一映射的用户值，其余重新询问。
- 用户中途修改流程：先确认切换，切换后重新获取字段；不要把旧流程字段直接带入新流程。

### 示例

用户说：“帮我发起请假，明天下午到后天下午，家里有事。”

假设当前日期为 2026-07-16，当前时区为 Asia/Shanghai。

期望处理：

1. 调用 `get-workflow-template-list`，若“请假申请”是唯一明显匹配的流程模板，则选定该模板；若存在多个请假相关模板，使用 `UserAsk` 让用户选择。
2. 调用 `get-workflow-template-form` 获取表单定义。
3. 将“明天”换算为 2026-07-17，将“后天”换算为 2026-07-18，并在后续摘要中展示绝对日期。
4. “下午”不是精确时间，若开始时间和结束时间字段要求 `yyyy-MM-dd HH:mm` 或具体时刻，必须使用 `UserAsk` 追问：
   - 开始时间：请提供 2026-07-17 下午的具体开始时间，格式如 13:30。
   - 结束时间：请提供 2026-07-18 下午的具体结束时间，格式如 18:00。
5. 若表单还有请假类型、时长等必填字段且无法从用户输入唯一确定，继续使用 `UserAsk` 补齐；不要自行默认年假、事假或 0.5 天等值。
6. 所有必填字段合法后，展示草稿摘要，例如：
   - 流程模板：请假申请
   - 开始时间：2026-07-17 13:30
   - 结束时间：2026-07-18 18:00
   - 请假原因：家里有事
   - 仍为空的可选字段数量：2
7. 询问用户是否确认发起。只有用户明确确认后，才调用 `create-workflow-by-template`；用户未确认、要求修改或取消时，不得提交。
