# 流程审批模块

## 当前范围

根据用户输入识别可发起的流程模板，得到流程模板 `ID`，读取对应表单的可填写字段，匹配用户提供的信息，并使用 `AskUserQuestion`
补全匹配失败的必填字段

## 发起审批

### 执行流程

#### 1. 解析流程模板

调用 `get-workflow-template-list` 工具， 获取当前用户可发起的模板。

并按以下优先级匹配：

1. `templateId` 完全一致
2. 模板名称完全一致
3. 用户描述与模板名称语义匹配

只在唯一候选明显成立时自动选择。零个候选时告诉用户没有找到可发起流程；多个候选时使用 `AskUserQuestion` 展示 2 至 5 个最相关候选

确认后保存 `templateId` 后续字段补全期间不要重复查询或更换流程，除非用户修改目标。

#### 2. 获取模板表单定义并建立草稿

调用 `get-workflow-template-form` 工具，并传入已确认的 `templateId`，获取模板对应表单定义。

遵循以下规则：

1. 告知用户需要优先填写的必填字段，尝试引导用户填写可选字段。
2. 识别当前会话用户的输入，自动填充主表字段和明细表字段草稿。
3. 严格以表单定义的字段为准，不要为用户猜测可能填写的字段。
4. 始终使用 `AskUserQuestion` 的方式引导用户输入确认字段值
5. 若返回的表单定义包含 `detailForms`，先根据明细表名称、字段标题和业务上下文判断用户是否提供了明细行，例如报销费用明细、出差行程、物品清单、用车记录等。每一条用户明确表达的记录应构造成一条
`detailDataList` 明细数据，明细数据的 `formId` 使用对应明细表的 `formId`，字段值放入该行的 `fieldDataList`。

明细表处理规则：

- 不要为了补齐明细表而自行创建空行；只有用户明确提供了一条记录，或某个明细表存在必填字段且流程无法不填时，才建立明细行或追问。
- 用户用顿号、逗号、分号、换行、序号、日期段、金额项等表达多条记录时，优先按自然分隔拆成多行；拆分后必须向用户展示可读摘要并等待最终确认。
- 每一行明细字段仍遵循通用字段值匹配规则和字段类型规则；同一行内字段缺失或歧义时，只追问该行必需的信息，不要把其他行的值复制为默认值。
- 明细表有多个候选时，必须能通过明细表名称、字段标题或业务上下文唯一确定；不能唯一确定时使用 `AskUserQuestion` 消歧。
- 明细表必填字段如果无法通过 Agent 填写，说明该明细表不支持通过 Agent 完成，并停止准备该流程。
- 明细表可选字段默认不追问，除非用户要求完整填写、该字段影响必填字段计算，或用户已经表达但存在格式/枚举歧义。
- 明细表中的金额、天数、数量等合计字段不要自行反推或改写，除非用户明确要求计算且目标字段语义唯一；计算结果必须在确认摘要中展示。
- 如果用户说“和上一条一样”“同上”等引用表达，只能复用用户已经明确确认过的字段值，并在摘要中标出复用后的显示值。

#### 3. 确认草稿

所有必填字段合法后，展示：

- 流程模板名称。
- 已填写字段的“字段标题：显示值”。
- 已填写明细表的“明细表名称：共 N 行”，并逐行展示关键字段的“字段标题：显示值”。
- 仍为空的可选字段数量。

使用 `AskUserQuestion` 询问用户是否确认发起。不要在确认摘要中展示内部字段 ID；提交工具仍必须使用字段 ID。

#### 4. 发起流程

只有用户确认后才调用名为 `create-workflow-by-template` 的工具。

正确参数结构示例：

```json
{
  "templateId": "<已确认的流程模板 templateId>",
  "values": {
    "formId": "<模板表单 formId>",
    "fieldDataList": [
      {
        "fieldId": "<字段 fieldId>",
        "fieldValue": "<已确认的字段值>"
      }
    ],
    "detailDataList": []
  }
}
```

除非用户要求，不要尝试重试调用

### 字段别名与业务映射

字段匹配除遵循通用字段值匹配规则外，审批类流程可使用以下常见别名组减少反复确认。别名组只能用于字段名称、字段标题或业务上下文明确命中的场景；若同一别名组命中多个候选字段，必须使用
`AskUserQuestion` 消歧。

| 用户表达或别名           | 可匹配字段含义                                             |
|--------------------------|------------------------------------------------------------|
| 事由、原因、说明、备注   | 申请原因、请假原因、加班原因、出差事由、报销事由、申请说明 |
| 开始、起始、从、开始时间 | 开始时间、开始日期、起始时间、起始日期                     |
| 结束、截止、到、结束时间 | 结束时间、结束日期、截止时间、截止日期                     |
| 时长、天数、小时数       | 请假时长、加班时长、出差天数、工时、用时                   |
| 类型、类别、假别         | 请假类型、报销类型、出差类型、加班类型                     |
| 金额、费用、花费         | 报销金额、费用金额、申请金额、合计金额                     |
| 地点、目的地、出差地     | 出差地点、目的地、城市、外出地点                           |
| 同行人、参与人、人员     | 同行人员、参与人员、相关人员                               |
| 项目、科目、用途、费用项 | 费用项目、报销科目、费用类型、用途                         |
| 数量、单价、票据、发票   | 数量、单价、票据张数、发票号码                             |

别名映射仍必须同时满足字段类型约束和枚举选项约束。例如用户说“事假”时，只有对应字段是请假类型且 `options`
中存在“事假”时才可自动匹配；不要把“家里有事”匹配到请假类型。

### 缺失字段追问策略

建立草稿后，立即找出所有仍缺失且可通过 Agent 填写的必填字段，包括主表必填字段和已建立明细行中的必填字段。第一次追问时尽量一次性询问最多
3 个缺失必填字段，不要一次只问一个字段。

追问优先级：

1. 会阻塞其他字段判断的必填字段，例如流程类型、请假类型、开始时间、结束时间。
2. 已建立明细行中会阻塞提交的必填字段，例如费用类型、金额、行程日期、出发地、目的地。
3. 用户已表达但存在精度或枚举歧义的字段，例如“明天下午”“报销交通费”。
4. 其他必填字段，例如原因、金额、时长、地点。

如果缺失必填字段超过 3 个，首轮只问优先级最高的 3 个；用户回答并校验通过后，再继续下一轮最多 3
个。可选字段默认不追问，除非用户要求完整填写或该字段影响必填字段计算。

### 错误边界

- 工具报权限错误：说明当前账号无权查询或发起，不要绕过权限查找其他接口。
- 流程模板在补全期间失效：丢弃草稿中的目标 ID，重新查询流程列表。
- 表单 schema 发生变化：以最新 schema 重新校验，保留仍可唯一映射的用户值，其余重新询问。
- 用户中途修改流程：先确认切换，切换后重新获取字段；不要把旧流程字段直接带入新流程。

### 示例要点

用户说“帮我发起请假，明天下午到后天下午，家里有事”时：

- 将相对日期换算为绝对日期并在摘要中展示。
- “下午”不是精确时刻；若字段要求具体时间，必须追问开始和结束具体时刻。
- 不要自行默认请假类型、时长或 0.5 天等值；缺失必填字段按追问策略一次最多问 3 个。
- 所有必填字段合法后展示草稿摘要，用户明确确认后才调用 `create-workflow-by-template`。
